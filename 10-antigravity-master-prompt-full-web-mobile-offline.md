## 🏗️ القرارات المعمارية غير القابلة للتفاوض

### 1. البنية العامة (Monorepo)
```
hamd/
├── apps/
│   ├── web/          # Next.js 14 — لوحة التحكم + API الرئيسي
│   └── mobile/        # React Native (Expo) — تطبيق البيع/المخزن Offline-First
├── packages/
│   ├── database/       # Prisma schema + client مشترك
│   ├── shared-types/    # أنواع TypeScript مشتركة بين web/mobile
│   └── business-rules/  # منطق حساب الأسعار/الضرائب/الخصومات (مشترك، ليُستخدم في validation على الطرفين)
```
استخدم **Turborepo** أو **pnpm workspaces** لإدارة الـ monorepo.

### 2. لماذا React Native (Expo) وليس PWA فقط للموبايل
PWA (`localStorage`/IndexedDB) غير موثوق بما يكفي للعمل الأوفلاين الحقيقي على المدى الطويل (قيود مساحة، سلوك غير متسق بين المتصفحات، لا وصول كامل لكاميرا/بلوتوث لطابعات الإيصالات). التطبيق الحقيقي يحتاج:
- قاعدة بيانات محلية حقيقية: **SQLite عبر `expo-sqlite`** (أو WatermelonDB لو الحجم كبير جدًا)
- وصول كامل للكاميرا لقراءة الباركود (`expo-camera` + `expo-barcode-scanner` أو `vision-camera`)
- طباعة عبر بلوتوث لطابعات إيصالات حرارية (`react-native-thermal-receipt-printer` أو مشابه)
- إشعارات Push حقيقية (Expo Notifications)

لوحة التحكم (Back-office) تبقى Next.js Web عادي (مش محتاجة أوفلاين لأنها للإدارة مش للبيع الميداني).

### 3. معمارية الأوفلاين/أونلاين (⭐ أهم جزء في المشروع)

#### أ) قاعدة البيانات المحلية (Mobile)
كل شاشة بيع/مخزون على الموبايل بتقرأ وتكتب من SQLite محلي **أولًا**، مش من السيرفر مباشرة. السيرفر هو "مصدر الحقيقة النهائي" لكن الموبايل بيشتغل بنسخة محلية محدّثة دوريًا.

#### ب) طابور المزامنة (Sync Queue)
كل عملية بتحصل أوفلاين (بيع، تحويل، جرد) بتتسجل في جدول محلي `SyncQueue` بالحالة:
```
PENDING → SYNCING → SYNCED | FAILED_CONFLICT
```
كل سجل في الطابور له `localId` (UUID يتولد على الجهاز وقت الإنشاء) عشان يبقى **Idempotent** — لو اتبعت مرتين بالغلط للسيرفر (بسبب انقطاع نت أثناء الإرسال)، السيرفر يتجاهل التكرار بناءً على `localId` (unique constraint على `referenceId` في `StockMovement`).

#### ج) استراتيجية حل التعارضات (Conflict Resolution)
هذا أخطر جزء ولازم يتنفذ بدقة:
1. **بيع أوفلاين لصنف نفد رصيده فعليًا على السيرفر** (باعه فرع تاني في نفس الوقت) → البيع **لا يُرفض تلقائيًا**، بل يُعلَّم كـ `NEEDS_REVIEW` ويظهر لمدير الفرع كتنبيه لحسمه يدويًا (بيع بالسالب مقصود، أو إلغاء، أو تحويل من فرع تاني) — **أبدًا لا تفقد الفاتورة ولا تلغيها تلقائيًا بدون علم المستخدم**.
2. **تعديل نفس السجل من جهازين مختلفين أوفلاين** (نادر لكن ممكن) → Last-Write-Wins بناءً على `createdAt` **مع الاحتفاظ بالنسخة الخاسرة في `AuditLog`** — لا حذف صامت لأي بيانات أبدًا.
3. عند عودة الاتصال، المزامنة تتم **بالترتيب الزمني** لكل جهاز (queue مرتب) وليس بالتوازي العشوائي، لضمان تسلسل حركات المخزون الصحيح.

#### د) مؤشر واضح للمستخدم
واجهة الموبايل لازم تُظهر دائمًا: 🟢 متصل ومتزامن / 🟡 أوفلاين — X عملية في الانتظار / 🔴 تعارض يحتاج مراجعة. المستخدم (الكاشير) لازم يعرف دايمًا إنه شغال أوفلاين، عشان يكون واعي.

### 4. الـ Stock Ledger (نفس المبدأ من ملف `02-stock-ledger-architecture.md`)
كل حركة مخزون — سواء جاية من الويب أو من الموبايل، أونلاين أو من طابور مزامنة أوفلاين — **تعدي حصريًا** من `StockMovementService` على السيرفر. الموبايل **لا يحسب** الرصيد النهائي بنفسه؛ بيحسب رصيد تقديري محلي للعرض فقط (من آخر نسخة متزامنة)، والرصيد الرسمي دايمًا من السيرفر.

### 5. Multi-tenancy + Tenant Isolation
كما هو مفصّل في `01-overview-and-tenancy-strategy.md`: Shared schema + `tenantId` إجباري في كل جدول + Prisma middleware يحقن الفلتر تلقائيًا + Postgres RLS كطبقة حماية ثانية. **لا استثناء لأي جدول عملياتي، بما فيه جداول المزامنة نفسها.**

### 6. الأمان
- JWT قصير العمر (15 دقيقة) + Refresh Token مخزّن بأمان (`expo-secure-store` على الموبايل، httpOnly cookie على الويب)
- كل API endpoint يتحقق من الصلاحية (RBAC) **على السيرفر**، لا تثق بأي فحص صلاحيات جاي من الـ client
- بيانات SQLite المحلية على الموبايل مشفّرة (SQLCipher) لو التطبيق بيتعامل مع بيانات مالية حساسة

## معايير القبول (Definition of Done) لأي كود تكتبه
1. كل عملية كتابة على المخزون تمر حصريًا من `StockMovementService` — لو لقيت نفسك بتكتب `UPDATE` مباشر على رصيد، توقف وراجع نفسك.
2. كل جدول عملياتي جديد فيه `tenantId` + index، وكل query فيه فلتر tenant (يدوي أو عبر middleware).
3. أي عملية أوفلاين على الموبايل لازم تكون Idempotent (قابلة للإرسال مرتين بدون أثر مضاعف).
4. لا حذف فعلي (`DELETE`) لأي سجل مالي أو مخزوني — Soft delete أو حركة عكسية فقط.
5. كل شاشة موبايل بتتعامل مع بيانات بيع/مخزون لازم تشتغل وهي أوفلاين تمامًا (اختبرها بقطع الـ WiFi فعليًا مش بس بالكود).
